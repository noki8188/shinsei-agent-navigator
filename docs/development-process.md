# 開発プロセスと agent の役割

## 要件定義

- 業務ヒアリング結果を申請ユースケースへ整理する
- actor、業務イベント、例外ケースを一覧化する
- 不明点を `Clarification` 観点として残す

## 基本設計

- 画面一覧と UI の責務を定義する
- 類型別データ項目と承認ルールを整理する
- ログ、監査、権限制御の必要点を列挙する

## 詳細設計

- 類型判定ロジックと必須項目の定義を詰める
- `PolicyDoc` `ClarificationItem` `DraftResult` `ReviewResult` を固定する
- 受入観点を UT / IT / ST に分けて設計する

## 開発

- Frontend は相談 UI と結果表示に集中する
- Backend は agent 役割ごとの責務分離を優先する
- Knowledge は Markdown で管理し、コードとレビューしやすくする

## テスト

- UT: 判定、補問、承認条件の単体確認
- IT: pipeline 全体の受け渡し確認
- ST: 業務シナリオでの妥当性確認

## リリース

- README を中心に背景と制約を明記する
- デモ手順を短く固定する
- 既知制約と次の拡張候補を整理する

## 運用保守

- 問い合わせ傾向から FAQ を増やす
- ルール変更時に knowledge を差し替えやすくする
- 人が確認した修正内容を次の改善サイクルへつなげる
